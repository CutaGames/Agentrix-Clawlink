## Bugs Fixed This Session

### 1. lib.rs file desync (CRITICAL)
- **Root cause**: The VS Code editor buffer had the correct 300+ line `lib.rs` with plugins, invoke_handler, and 20 command wrappers, but the **on-disk file was a stale 141-line skeleton** with no plugins, no commands, and a bare `Builder::default().run()`. Every `cargo build` compiled the stale version.
- **Evidence**: Exe was only 10.2 MB (missing 7 MB of plugins). ALL commands returned "Command not found".
- **Fix**: Wrote correct content to disk. Exe grew to 17.4 MB.

### 2. CSP blocking Tauri IPC (CRITICAL)
- **Root cause**: Content Security Policy's `connect-src` directive did not include `http://ipc.localhost`. Tauri 2 uses `fetch("http://ipc.localhost/cmd")` as its primary IPC transport. CSP blocked the fetch, and the postMessage fallback also failed.
- **Evidence**: Console warning: `"IPC custom protocol failed, Tauri will now use the postMessage interface instead TypeError: Failed to fetch"`. Commands hung (Promise never resolved/rejected).
- **Fix**: Added `http://ipc.localhost` to CSP `connect-src` in `tauri.conf.json`.

### 3. open_chat_panel main-thread deadlock (HIGH)
- **Root cause**: `open_chat_panel` was a sync command calling `WebviewWindowBuilder::build()`, which internally sends a message to the main thread event loop and waits. Since the command itself was blocking the main thread, this caused a deadlock.
- **Evidence**: Invoking `open_chat_panel` hung forever, and all subsequent commands on the same connection also hung.
- **Fix**: Changed `open_chat_panel` to fire-and-forget window creation via `std::thread::spawn`.

### 4. build.rs APP_COMMANDS name mismatch (MEDIUM)
- **Root cause**: `build.rs` declared old command names (e.g., `open_chat_panel`) while actual functions use `desktop_bridge_*` prefix.
- **Fix**: Updated all 20 names in `APP_COMMANDS`. Autogenerated permission TOMLs regenerated correctly.
