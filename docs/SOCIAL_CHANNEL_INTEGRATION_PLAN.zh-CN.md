# Agentrix 社交渠道接入方案

生成时间：2026-03-16
负责人：GitHub Copilot 审计补充

关联总纲：docs/UNIFIED_AGENT_PRESENCE_VISION.zh-CN.md

## 1. 目标

这里要先纠正一个产品视角。

社交监听和渠道接入不是为了把 Agentrix 做成单纯的“社交客服工具”，而是为了让用户拥有的某个选定 Agent 在不同硬件、不同应用、不同社交渠道中持续陪伴用户，形成统一的 Agent Presence。

因此真正目标应改写为：

建设统一的 Agent Presence 层，让用户无论是在移动端、桌面端、Telegram、飞书、企业微信、Discord、Twitter/X，还是在语音免看屏场景里，都能随时随地与自己选定的某个 Agent 协作，而且该 Agent 的数据、上下文、记忆、偏好和任务状态保持共享。

同时要明确：

- 一个用户可以拥有多个 Agent
- 每个 Agent 可以是不同人格、不同职责、不同记忆边界的分身
- Agent 之间是否共享记忆、知识库、技能、联系人和上下文，必须由用户自己决定

这个目标下，社交通讯层只是 Agent Presence 的一个子系统，需要统一完成以下流程：

- 接收外部社交消息
- 绑定到同一个用户 Agent / 工作区 / 任务上下文
- 与移动端和桌面端会话数据打通
- 按策略自动回复、审核发送、语音处理或仅通知
- 在必要时由 Bot 代表用户参与渠道互动维护
- 记录完整会话、送达状态、记忆写入与操作审计

本文档覆盖：

- 优先支持哪些渠道
- 在总目标下每类渠道承担什么角色
- 渠道接入后的端到端流程
- 客户需要提供什么信息
- Agentrix 还需要补哪些能力
- 怎样把体验做成真正的跨硬件协作
- 移动端、桌面端、Bot 各自应该承担什么角色

## 2. 北极星定义

未来产品应围绕下面这句定义来设计：

“用户拥有一个持续在线的个人 Agent，它可以跨设备、跨应用、跨社交渠道与用户协作，并在得到授权后代替用户处理一部分沟通、维护和执行任务。”

这意味着必须满足 5 个基础条件：

1. 同一个 Agent 身份：不是每个渠道一个独立机器人，而是用户选定的某个 Agent 在多个入口显化
2. 同一份会话与记忆：移动端、桌面端、Telegram、飞书等看到的是该 Agent 的统一上下文
3. 同一份任务状态：用户在手机上对某个 Agent 说一半的话，桌面端和社交 Bot 能接着做
4. 支持免看屏语音：重要信息能被读出来、总结出来、语音确认处理
5. 支持代理参与：在明确授权边界内，Bot 可以代表用户维护部分社交互动
6. 支持多 Agent 自主管理：用户能决定 Agent 之间共享什么，不共享什么

## 3. 当前状态总结

当前代码库里实际存在两条路线：

### A. Social Listener / Social Bridge

作用：

- 接收 Telegram、Discord、Twitter/X 回调
- 落库保存事件
- 在移动端展示连接状态、审核队列和回复策略

当前状态：

- Telegram、Discord、Twitter/X 的入站回调接口已存在
- 社交事件持久化已存在
- 回复策略配置已存在
- 移动端 Social Listener 页面已存在
- 出站回复闭环仍然不完整，尤其是跨平台复用能力不足

结论：

- 这条线适合作为管理后台和审核控制台基础
- 还不能算完整的“客户可用社交客服桥接层”

### B. Telegram Bot 直连 OpenClaw

作用：

- 通过 Telegram deep link / token 把聊天绑定到 OpenClaw 实例
- 把消息转发到本地、平台托管或外部 Agent 运行时
- 再把回复发回 Telegram

当前状态：

- Telegram 绑定链路已存在
- 消息转发已存在
- 语音转写已存在
- 回复回发已存在

结论：

- 这条线更接近客户可直接使用的真实闭环
- 但目前只覆盖 Telegram，而且和 Social Listener 的数据模型还没有完全统一

## 4. 推荐的产品方向

长期不应该继续维护两套平行社交架构。

建议统一成一条更高层的 Agent Presence 标准链路：

1. Agent Identity 层
2. 渠道连接层
3. 身份绑定与权限层
4. 事件归一化层
5. 单 Agent 会话与记忆层
6. 路由与任务编排层
7. 多 Agent 共享与隔离策略层
8. 出站发送层
9. 审计与会话历史层
10. 移动端 / 桌面端 / 语音 / Bot 统一协作界面

原则：

- 外部渠道各不相同，内部数据结构必须统一
- 内部统一成 Agent Session + Conversation Event + Delivery Action
- 社交消息不是孤立消息，而是某个 Agent 统一时间线的一部分
- 任何渠道消息都应该可以写入该 Agent 的长期记忆、短期上下文和任务状态
- 是否进一步共享给其他 Agent，必须走用户配置的共享策略

## 5. 渠道优先级

这里的优先级不是“哪个客服渠道先接”，而是“哪个入口最能帮助某个选定 Agent 覆盖用户真实日常工作流”。

### 第一优先级

#### Telegram

适用场景：

- 高频私聊协作
- 随时随地与某个指定 Agent 对话
- 海外个人工作流入口
- 语音消息入口
- Bot 代管部分社群维护

接入方式：

- Bot Token + Webhook
- 可选绑定到指定 Agent / 工作区 / OpenClaw 实例

当前结论：

- 应优先做成第一条真正闭环的 Agent Presence 入口

#### Discord

适用场景：

- 社区运营
- 支持频道
- 角色权限驱动自动化
- Slash Commands

接入方式：

- Discord Application + Bot + Interaction Endpoint

当前结论：

- 回调骨架已有
- 适合在 Telegram 闭环稳定后复用同一策略和发送框架

#### Twitter / X

适用场景：

- 品牌提及监控
- 私信分流
- 活动回复

接入方式：

- 开发者应用 + 活动 API / Webhook

当前结论：

- 适合作为“监听 + 草稿 + 人审”的高优先级渠道

### 第二优先级：国际高价值渠道

- WhatsApp Business：适合全球商务支持和线索转化
- Slack：适合 B2B 内部协作和企业审批流

### 第三优先级：中国企业渠道

- 飞书：最适合中国区企业协作、消息流转和 Agent 工作台联动
- 企业微信：最适合企业内部沟通和客户服务并存场景
- 钉钉：适合流程审批、组织任务和企业执行链路
- QQ：可覆盖 C 端，但企业 API 能力弱，优先级应低于飞书和企微

## 6. 标准接入流程

所有平台都应遵循同一产品流程：

### 第 1 步：选择渠道

客户选择：

- Telegram
- Discord
- Twitter/X
- WhatsApp
- Slack
- 飞书
- 企业微信
- 钉钉

### 第 2 步：选择连接模式

支持三种模式：

- 客户自有官方 Bot / App
- Agentrix 托管共享 Bot，用于快速开始
- Agentrix 专属托管 Bot，用于付费客户

建议：

- 快速体验走 Agentrix 托管共享 Bot
- 正式生产走客户自有或专属托管 Bot

### 第 3 步：绑定目标

客户选择路由目标：

- 一个用户个人 Agent
- 一个 OpenClaw 实例
- 一个团队工作区
- 一个具体任务或工作流
- 一个支持队列

### 第 4 步：配置路由规则

客户配置：

- 只接 DM 还是 DM + Channel
- 只响应 mention 还是所有消息
- 语言分流
- 部门分流
- VIP 分流
- 升级人工规则

### 第 5 步：配置回复策略与代理级别

支持策略：

- 自动回复
- 审核后发送
- 仅通知
- 完全禁用

还应补充代理级别：

- 观察模式：只监听和总结
- 助理模式：生成草稿，等用户确认
- 代表模式：在授权边界内自动参与互动
- 维护模式：定期维护频道、社群和私聊关系

### 第 6 步：测试、语音验证与上线

客户完成：

- Webhook 检测
- 授权校验
- 测试消息
- 策略验证
- 语音播报与语音确认验证
- 上线确认

## 7. 客户需要提供什么

### Telegram

- Bot Token
- Bot 用户名
- 是否绑定到某个 Agent / 实例

### Discord

- Application / Bot
- Public Key
- Client ID
- Bot Token
- 服务器或频道范围

### Twitter / X

- 开发者账号
- App Key / Secret
- Access Token / Secret
- 回调配置

### 企业渠道通用补充信息

- 接入主体归属
- 是否需要审计留痕
- 是否允许自动回复
- 是否允许 Agent 代表用户发言
- 哪些类型的信息允许免看屏语音处理
- 工作时间与静默时间
- 升级人工联系人
- 默认语言与语气风格
- 需要共享给 Agent 的频道、群组、联系人范围

## 8. Agentrix 还需要建设的能力

### A. 统一会话与记忆层

这是最关键的一层，必须补：

- 单个 Agent 的跨端统一会话 ID
- 社交消息、移动端聊天、桌面端聊天统一写入会话时间线
- 短期上下文和长期记忆共享
- 用户偏好、关系状态、频道语气记忆共享

### B. 免看屏语音能力

需要补成真正的产品能力，而不是单独的语音输入组件：

- Agent 能朗读社交摘要和待处理事项
- 用户可以用语音说“回复他”“先归档”“帮我总结重点”
- 对高优先级消息支持耳机 / 车载 / 穿戴设备场景处理
- 对免看屏场景必须有安全边界，避免误发消息

### C. Agent 代表用户参与互动

需要明确授权分级：

- 哪些场景只能生成草稿
- 哪些场景可以自动回复
- 哪些群组允许 Agent 代维护
- 哪些账号与平台禁止自动发言

### D. 统一运营台

移动端、桌面端和 Bot 不是三个孤立产品，而是同一个运营系统的不同界面。

### 必做

- 统一事件模型与统一出站发送层
- 统一策略引擎
- 统一审核队列
- 统一送达状态追踪
- 统一会话历史与搜索
- 统一失败重试与告警
- 统一 Agent 记忆写入与检索
- 统一跨端会话同步

### 体验增强

- 渠道接入向导
- 一键复制回调地址
- 渠道健康检查
- 测试消息向导
- FAQ 与错误诊断

### 后台能力

- 审计日志
- 运营数据面板
- 自动分类与路由
- VIP / 风险 / 黑名单规则

## 9. 为什么桌面端必须参与

桌面端不只是“另一个入口”，它更适合做单个 Agent 的长期工作台，以及多 Agent 的协同管理台：

- 多会话并发处理
- 审核草稿批量操作
- 渠道状态监控
- 快捷键处理消息
- 大屏查看会话上下文、知识库、Agent 日志
- 挂机运行、保持 Agent 常驻
- 和企业办公软件协同工作

建议定位：

- 移动端：随时对话、语音处理、快速审批
- 桌面端：正式运营、批量处理、深度配置、持续在线工作台
- Bot / 渠道端：用户不打开 Agentrix 也能继续和某个指定 Agent 协作

## 10. “免看屏”和“代表参与”的边界

并不是所有社交信息都适合语音和自动代理处理，建议分层：

### 可优先支持免看屏语音处理

- 私聊新消息摘要
- 会议提醒、任务提醒、待审批提醒
- 用户说一句话生成简短回复
- 语音确认“发送”“稍后提醒我”“加入待办”

### 适合 Agent 代表用户参与的场景

- 固定 FAQ 回复
- 社群欢迎语、基础维护
- 已知流程类回复
- 跟进未回复消息
- 定期内容分发和轻运营互动

### 不应默认自动代理的场景

- 涉及合同、付款、法律和高风险承诺
- 涉及对外立场表达
- 涉及敏感个人关系沟通
- 涉及复杂谈判

## 11. 分阶段路线图

### Phase 1

- 打通 Telegram 真闭环
- 打通社交消息和主聊天会话的统一时间线
- 支持四种回复策略
- 支持语音读出摘要与语音确认发送
- 让移动端和桌面端都能看到同一批事件、记忆和状态

### Phase 2

- 抽象统一发送接口
- 复用到 Discord
- 复用到 Twitter/X
- 增加 Agent 代表用户参与的授权分级

### Phase 3

- 接入 WhatsApp / Slack / 飞书 / 企微
- 增加运营分析、告警、自动分类
- 增加桌面端常驻工作台和多渠道工作流

## 12. 当前建议结论

短期最正确的做法不是推翻现有代码，而是：

1. 先把 Telegram 做成第一个“单个 Agent 跨端持续在线”的闭环入口
2. 把社交消息统一并入该 Agent 的主会话、主记忆和主任务状态
3. 在移动端先补“语音摘要 + 语音确认处理”链路
4. 再补用户控制的多 Agent 共享与隔离策略
5. 再把同一套策略、记忆、发送框架复用给 Discord、Twitter/X、飞书等渠道
6. 最后把桌面端做成多 Agent 的正式工作台，而不是单纯社交后台
