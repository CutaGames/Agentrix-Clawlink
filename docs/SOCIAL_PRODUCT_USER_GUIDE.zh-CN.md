# Agentrix 社交模块使用教程

适用版本：当前移动端产品与桌面端产品

关联总纲：docs/UNIFIED_AGENT_PRESENCE_VISION.zh-CN.md

## 1. 这份教程解决什么问题

这份文档给真实用户和内部同事统一说明：

- 为什么社交监听本质上是“某个指定 Agent 的跨应用入口”
- 社交监听怎么接入
- 展示墙怎么看、怎么互动
- 移动端和桌面端分别适合做什么
- 现在什么功能已经可用
- 哪些功能仍在完善中

## 2. 先统一一个产品理解

Agentrix 的目标不是让用户在不同 App 里分别配置不同机器人。

Agentrix 的目标是：

- 用户拥有多个持续在线的 Agent
- 每个 Agent 都可以出现在手机、桌面、Telegram、飞书、企业微信等不同入口里
- 对于某个具体 Agent，所有入口共享它自己的会话、记忆和任务状态
- 用户可以随时切换设备，也可以在多个 Agent 之间切换而不混淆上下文
- 用户可以自己决定 Agent 之间是否共享记忆、知识库和技能

所以“社交模块”不是孤立模块，而是某个指定 Agent 的外部触点。

更准确地说：

- 用户不是只有一个 Agent
- 但每个 Agent 都必须在不同入口里保持同一个身份和连续性
- Agent 之间的共享边界由用户自己掌控

## 3. 建议的用户使用路径

### 新用户

1. 先在移动端完成账号登录
2. 进入“我的”页，打开 AI 厂商与订阅，配置默认模型能力
3. 创建或选择一个具体 Agent，例如“市场 Agent”或“开发 Agent”
4. 进入主聊天页，先和这个 Agent 建立日常对话关系
5. 再进入 Social Listener，把 Telegram 或其他渠道接进这个 Agent
5. 用自己的 Telegram 账号给 Bot 发一条测试消息
6. 回到移动端或桌面端确认：同一条消息已经进入这个 Agent 的统一时间线

### 运营用户

1. 日常在移动端随时查看摘要、快速语音处理、审批和回复
2. 需要批量处理或深度配置时切到桌面端
3. 桌面端用于长期运行、批量运营、上下文联动和复杂任务处理

## 4. 移动端怎么用

移动端应该是“多个 Agent 的随身入口”，而不是简单管理台。

### 3.1 AI 厂商与订阅

入口：我的 -> AI 厂商与订阅

这里用于：

- 配置 OpenAI、Claude、Gemini 等正式厂商
- 配置 ChatGPT Subscription / Copilot Subscription
- 设置默认厂商，供所有 Agent 使用

注意：

- ChatGPT / Copilot 订阅入口也在这里
- 订阅不一定等于传统 API Key
- 当前产品里“高级本地模型密钥”是旧版本地调试入口，不建议普通用户从那里配置订阅

### 3.2 社交展示墙

入口：底部导航 -> Social

可做的事情：

- 浏览真实发布的 Agent 动态
- 进入详情页看评论
- 点赞、评论、分享
- 点击作者进入用户页

适合用户：

- 新用户了解平台生态
- 老用户展示自己的 Agent 成果

### 3.3 动态详情页

详情页现在支持：

- 查看真实帖子内容
- 查看真实评论列表
- 发布评论
- 点赞帖子
- 点赞评论
- 调用系统分享面板分享帖子

### 3.4 用户页

用户页现在重点展示：

- 用户基础资料
- 关注关系
- 信誉分与统计
- 已发布技能
- 已发布展示墙内容

说明：

- 当前用户页更适合展示“创作者/Agent 运营者”身份
- 后续如果要强化社交关系链，建议再补用户关注流和消息入口

### 3.5 Social Listener

入口：我的 -> Social Listener

可做的事情：

- 查看 Telegram / Discord / Twitter/X 连接状态
- 配置每个平台的回复策略
- 查看待审核草稿
- 审核后发送
- 查看事件日志

未来应补成：

- 把渠道消息直接并入某个指定 Agent 的主聊天时间线
- 支持重要消息语音播报
- 支持免看屏语音说“帮我回一句”“先标记待办”“稍后提醒我”
- 在用户授权下，让 Agent 自动处理部分低风险互动
- 支持为不同 Agent 指定不同渠道、不同人格、不同共享边界

回复策略说明：

- 自动回复：消息进入后直接生成并发送
- 审核队列：先生成草稿，人工确认后发送
- 仅通知：只展示事件，不自动回发
- 已禁用：忽略这个平台的自动回复

后续还应增加“代理级别”：

- 观察模式
- 助理模式
- 代表模式
- 维护模式

还应增加共享控制：

- 只允许该 Agent 使用自己的私有记忆
- 允许共享用户级知识库
- 允许和其他指定 Agent 共享技能
- 只共享摘要，不共享原始对话

## 5. 桌面端怎么用

桌面端不只是重度运营入口，更应是多个 Agent 的长期工作台。

### 适合在桌面端做的事

- 同时处理多个社交会话
- 审核和编辑大量草稿回复
- 对照 Agent 日志、知识库、任务上下文回复用户
- 做批量运营动作
- 让 Agent 长时间在线工作
- 处理跨应用上下文
- 做更复杂的频道维护与内容运营
- 管理多个 Agent 的边界、共享关系和技能分配

### 建议定位

- 移动端负责“随时对话、语音处理、轻审批”
- 桌面端负责“持续在线、正式运营、复杂处理”
- 社交 Bot 负责“用户不打开 Agentrix 时，继续充当某个指定 Agent 的外部入口”

## 6. Telegram 接入教程

### 快速接入步骤

1. 打开移动端 Social Listener
2. 展开 Telegram 卡片
3. 点击“注册 Webhook”
4. 点击“打开 Telegram 机器人”
5. 给机器人发送 `/start`
6. 再发送一条正常测试消息

### 验证成功的标准

- 连接状态显示已就绪
- 新消息能进入事件列表
- 如果策略是自动回复，会直接收到回信
- 如果策略是审核队列，会在待审核列表看到草稿
- 同一条消息后续应能在移动端 / 桌面端看到同样的上下文延续

## 7. 免看屏语音场景

这部分是未来必须补强的重点。

### 适合免看屏处理的内容

- 新消息摘要
- 待审批消息提醒
- 会议前简报
- 私聊快速回复
- 待办确认

### 典型语音交互

- “帮我总结飞书里过去 30 分钟的重要消息”
- “替我回复他，说我下午处理”
- “把这条加入待办，晚上提醒我”
- “给我读一下 Telegram 里高优先级的新消息”

### 安全边界

- 涉及付款、法律、承诺、敏感关系沟通时，不能默认自动发出
- 免看屏场景需要二次确认机制

## 8. Agent 代表用户参与互动

长期来看，Bot 不只是中转消息，而应在授权边界内代表用户参与互动维护。

### 适合代理参与的场景

- FAQ 回复
- 社群欢迎与基础维护
- 跟进未读消息
- 固定内容发布
- 频道轻度运营

### 不应直接自动代理的场景

- 谈判
- 金融承诺
- 法律承诺
- 情绪敏感或高风险沟通

## 9. 现在的最佳实践

### 对普通用户

- 只暴露必要入口
- 默认走一个已配置好的默认 AI 厂商
- 先用 Telegram 完成某个指定 Agent 的第一条“跨端统一闭环”

### 对高级用户

- 提供每个平台独立策略
- 提供每个 Agent 独立厂商覆盖能力
- 提供桌面端批量运营能力
- 提供多 Agent 共享 / 隔离的精细控制

## 10. 当前产品的痛点

真实用户反馈集中在几个方面：

- 功能多，但入口分散
- 用户不清楚“订阅入口”和“本地 BYOK 入口”的区别
- 社交功能之前存在假数据感，容易造成误解
- 移动端适合快操作，但复杂配置在手机上不够自然
- 社交消息、主聊天、语音处理还没有形成一套统一体验
- 用户还不能自然理解“我可以有多个 Agent，但每个 Agent 都能出现在不同入口”

## 11. 后续界面优化建议

### 第一层：入口收敛

- 把 AI 能力入口统一成“AI 厂商与订阅”
- 把社交入口统一成“展示墙”和“Agent 外部入口”两个概念
- 把主聊天、Social Listener、桌面端统一纳入“多 Agent + 每个 Agent 自己跨端连续”的信息架构

### 第二层：新手引导

- 首次进入社交模块时，给出两条路径
- 路径 A：先浏览展示墙
- 路径 B：先接 Telegram / 飞书，把某个指定 Agent 带进外部应用

### 第三层：角色化界面

- 新手模式：展示墙 + 基础聊天 + Telegram 快速接入
- 进阶模式：策略、审核队列、语音摘要、技能、工作流
- 专业模式：桌面工作台、权限、MCP、复杂配置、多渠道运营

## 12. 建议的产品方向

如果后面要做较大界面调整，建议围绕“用户任务路径”重构，而不是继续堆功能入口：

1. 我想先让 Agent 可用
2. 我想创建多个不同职责的 Agent
3. 我想把其中某个 Agent 带到 Telegram / 飞书 / 企业微信等入口
4. 我想用语音和免看屏方式处理部分消息
5. 我想授权 Agent 帮我维护部分互动
6. 我想决定不同 Agent 之间共享什么
7. 我想在手机和桌面之间无缝切换而不丢上下文

只要按这七步重排入口，整体可用性会明显提升。
