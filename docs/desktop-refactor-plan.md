# Agentrix Desktop 重构优化计划

**日期**: 2026-04-01  
**状态**: 待审核  
**核心问题**: 桌面端产品定位偏移——既不像轻量语音助手，也不像专业Agent工作台

---

## 一、现状诊断

### 1.1 截图问题分析

从当前截图可以看到：
- 聊天面板显示 "Runtime Timeline" + 文件读取操作列表（ Read file, List directory）
- 底部是 Ask / Agent / Plan 三个模式切换
- 整体UI更像 **VS Code Copilot Chat 的翻版**，不是一个消费级语音助手
- 语音模式下没有语音特有的交互体验，只是在同一个面板里显示文字
- 复杂任务模式下显示的都是底层操作细节（读文件、列目录），用户无法理解

### 1.2 与 heylemon.ai 的差距

| 维度 | heylemon.ai | Agentrix 当前 | 差距 |
|------|-------------|--------------|------|
| **核心交互** | 按fn键→说话→完成 | 点悬浮球→展开面板→打字/录音 | 太多步骤 |
| **语音体验** | 随时唤醒，零摩擦 | 需要展开面板才能看到结果 | 语音不是一等公民 |
| **界面形态** | 最小化，不打断工作流 | 480×640大面板，遮挡工作 | 太重 |
| **结果呈现** | 直接执行（回复邮件、创建文档） | 显示代码执行时间线 | 面向开发者非用户 |
| **上下文感知** | 自动识别当前应用/选中文本 | 需要手动输入或拖拽文件 | 不够智能 |

### 1.3 根本问题：身份危机

当前桌面端试图同时做两件事:
1. **轻量语音助手**（像 Siri/heylemon）—— 但语音交互被塞进了重型面板
2. **专业Agent IDE**（像 VS Code Copilot）—— 但缺少代码diff、终端输出等关键能力

结果两边都做不好：**语音场景太复杂，专业场景太简陋**。

## 三、语音模式（Voice Mode）— 参考 heylemon.ai

### 3.1 交互流程

```
平时状态:
  悬浮球安静地待在屏幕边缘（半隐藏）

唤醒方式（任选）:
  ① 单击悬浮球
  ② 全局热键 Ctrl+Shift+A
  ③ 语音唤醒词 "Hey Agentrix"
  ④ 长按悬浮球（直接进入录音）

唤醒后 → 悬浮球变形:
  ┌─────────────────────────────────────┐
  │  ◉ ≋≋≋≋≋≋≋≋  正在聆听...            │  ← 胶囊形态，显示波形
  └─────────────────────────────────────┘

用户说话 → 实时转写显示:
  ┌─────────────────────────────────────┐
  │  ◉ ≋≋≋≋  "帮我查一下今天上海天气"    │  ← 实时显示说话内容
  └─────────────────────────────────────┘

松开/停顿 → 处理中:
  ┌─────────────────────────────────────┐
  │  ◉ ⟳ 正在思考...                     │  ← 旋转动画
  └─────────────────────────────────────┘

回复到达 → 语音播报 + 文字卡片:
  ┌─────────────────────────────────────────────────┐
  │  ◉ 🔊                                          │
  │  ┌───────────────────────────────────────────┐  │
  │  │ 上海今天 23°C，多云转晴，东风3-4级           │  │  ← 浮动结果卡片
  │  │ 明天降温到 18°C，建议带外套                  │  │     4秒后自动收起
  │  └───────────────────────────────────────────┘  │
  └─────────────────────────────────────────────────┘

播报完毕 → 自动收回悬浮球 (或继续对话)
```

### 3.2 语音模式 UI 规格

| 项目 | 规格 |
|------|------|
| **形态** | 不打开任何面板窗口，所有交互在悬浮球上完成 |
| **输入** | 语音为主，点击胶囊可切换到文字快捷输入 |
| **输出** | TTS语音播报 + 浮动结果卡片（最大宽度 400px，自适应内容高度） |
| **结果卡片** | 简洁的回答摘要，非完整Markdown渲染 |
| **持续对话** | 回复后悬浮球保持活跃5秒，用户可继续说话 |
| **退出** | 5秒无交互自动收回 / Esc键 / 点击其他区域 |
| **流式播报** | Agent回复边生成边播报（分句TTS），不等全部完成 |
| **快捷操作** | 对结果卡片右键：复制、展开详情（自动切换到专业模式）、重新回答 |

### 3.3 语音模式核心实现变更

**当前**: 语音录音 → 展开ChatPanel → 显示在聊天消息列表里
**重构**: 语音录音 → 在悬浮球原位展示结果卡片 → TTS播报 → 自动收回

关键改造:
1. **FloatingBall.tsx** — 增加结果卡片层（不开新窗口）
2. **VoiceButton.tsx** — 合并到FloatingBall内部，不再是独立组件
3. **新增 VoiceResultCard.tsx** — 简洁的浮动结果卡片
4. **voice.ts** — 增加分句流式TTS（已有AudioQueuePlayer基础）
5. **主窗口** — 从400×400调整为自适应大小（最小56×56，扩展时最大460×auto）

## 四、专业模式（Pro Mode）— 增强版Agent工作台

### 4.1 入口方式

```
唤醒专业模式:
  ① 双击悬浮球
  ② Ctrl+Shift+S 热键
  ③ 语音说 "打开工作台" / "专业模式"
  ④ 语音模式下结果卡片点击 "展开详情"
  ⑤ 系统托盘右键 → "打开工作台"
```

### 4.2 专业模式重构要点

当前 ChatPanel 的问题是想做VS Code Copilot但深度不够。重构方向:

#### A. 分层展示（不再一股脑显示底层操作）

```
┌──────────────────────────────────────────┐
│ 🤖 Claude Sonnet 4.6  ▼模型  📂项目  ⚙  │
├──────────────────────────────────────────┤
│                                          │
│  🧑 帮我重构 auth 模块的登录逻辑          │
│                                          │
│  🤖 好的，我来帮你重构登录逻辑。           │
│                                          │
│  📋 执行计划                    [展开 ▼]  │  ← 折叠的计划卡片
│  ├─ ✅ 分析现有代码结构                   │
│  ├─ ⏳ 重构 auth.service.ts              │
│  ├─ ○ 更新测试用例                        │
│  └─ ○ 验证编译通过                        │
│                                          │
│  📝 正在修改 auth.service.ts...           │  ← 当前步骤摘要
│  ┌────────────────────────────────┐      │
│  │ - const token = jwt.sign()    │      │  ← 代码diff预览
│  │ + const token = await ...     │      │
│  └────────────────────────────────┘      │
│                                          │
├──────────────────────────────────────────┤
│ ┌──────────────────────────┐ 🎤 📎 ➤   │
│ │ 输入消息...               │            │
│ └──────────────────────────┘             │
│  Ask   Agent   Plan                      │
└──────────────────────────────────────────┘
```

**核心变化**: Runtime Timeline 底层操作列表改为**用户可理解的执行计划视图**:
- 默认折叠底层操作（Read file、List directory 等）
- 显示高层任务进度（分析→修改→测试→验证）
- 代码变更用diff视图而非文件名列表
- 只在"详细"模式下展开底层操作

#### B. 真正可用的模式切换

| 模式 | 用途 | 当前状态 | 重构目标 |
|------|------|---------|---------|
| **Ask** | 快速问答 | ✅ 基本可用 | 保持，优化响应速度 |
| **Agent** | 自动执行任务 | ⚠️ 显示底层操作 | 分层展示+计划审批 |
| **Plan** | 先规划后执行 | ❌ 不可用 | 接入PlanPanel，可编辑计划 |

#### C. 缺失的专业能力补齐

| 能力 | 优先级 | 说明 |
|------|--------|------|
| 代码Diff渲染 | P0 | diff2html，修改文件时显示前后对比 |
| 终端输出流式显示 | P0 | 命令执行结果实时显示，ANSI颜色支持 |
| Thinking动画指示器 | P0 | "思考中 (3.2s)" 而非静态文本 |
| 计划审批UI | P1 | PlanPanel已有但未接入 |
| Token/上下文显示 | P1 | 输入框下方显示token消耗进度条 |
| 多Agent工作流 | P2 | 可视化Agent间协作 |

## 六、PRD 更新建议

### 需要更新的部分:

#### 6.1 产品定位更新

```diff
- 桌面端 AI Agent 入口 — 悬浮球 + 语音唤醒 + 跨设备协同
+ 桌面端 AI Agent 入口 — 语音优先的双模交互（Voice Mode + Pro Mode）
```

增加核心理念:
> **语音模式是默认模式**，90%的日常交互在悬浮球上完成，无需打开任何面板。
> 专业模式是深度工作的补充，处理需要视觉反馈的复杂任务。

#### 6.2 悬浮球规格更新

当前PRD的悬浮球规格大体正确，但缺少:
- 结果卡片浮窗（语音回复不应该打开ChatPanel）
- 持续对话窗口期（回复后保持活跃等待追问）
- 自适应窗口大小（从56×56到最大460×auto）
- 双击触发专业模式

#### 6.3 新增语音模式单独章节

PRD中语音是ChatPanel的附属功能，应该独立为一等公民:
- 语音模式有独立的UI状态机
- 语音模式的结果展示方式不同于文字聊天
- 语音模式的会话是轻量且短暂的

#### 6.4 对话面板改名为"专业工作台"

```diff
- 4.2 对话面板 (Chat Panel)
+ 4.2 专业工作台 (Pro Workspace)
```

明确定位为深度工作工具，而非语音回复的附属显示面板。

#### 6.5 里程碑更新

```diff
- P0-Alpha | Week 1-2 | Tauri 项目骨架 + 悬浮球 + 基础对话面板
+ Phase 1  | Week 1-2 | 语音模式重塑 — 悬浮球语音交互达到 heylemon 水准
+ Phase 2  | Week 3-4 | 专业模式升级 — 任务可视化 + Diff + 计划审批
+ Phase 3  | Week 5   | 上下文智能 — 自动感知选中文本/活动窗口
+ Phase 4  | Week 6   | 打磨完善 — Token显示 + 性能优化 + 引导流程
```

## 七、技术要点

### 7.1 悬浮球窗口自适应

当前 `tauri.conf.json` 固定 400×400，需要改为:
- 默认: 56×56 (只显示球体)
- 胶囊态: 460×56 (录音+波形+转写)
- 结果态: 460×auto (球体+结果卡片，最大高度300px)
- 通过 Rust `set_size()` 动态调整，而非创建新窗口

```rust
// lib.rs 新增
#[tauri::command]
fn desktop_bridge_resize_ball_window(app: AppHandle, width: f64, height: f64) {
    if let Some(win) = app.get_webview_window("main") {
        let _ = win.set_size(tauri::Size::Logical(LogicalSize { width, height }));
    }
}
```

### 7.2 语音结果不走ChatPanel

```typescript
// FloatingBall.tsx 语音流程
const handleVoiceResult = async (transcript: string) => {
  setBallState("thinking");
  
  // 直接流式请求，不通过ChatPanel
  const abort = streamDirectChat({
    messages: [{ role: "user", content: transcript }],
    sessionId: voiceSessionId,
    token,
    onChunk: (chunk) => {
      setResultText(prev => prev + chunk);
      // 分句TTS
      audioQueue.feedChunk(chunk);
    },
    onDone: () => {
      setBallState("speaking");
      // 5秒后自动收回
      setTimeout(() => setBallState("idle"), 5000);
    }
  });
};
```

### 7.3 分层任务展示

```typescript
// TaskTimeline.tsx 重构
// 当前: 平铺显示所有底层操作
// 重构: 高层摘要 + 可展开详情

interface TaskStep {
  id: string;
  label: string;              // "分析代码结构" 而非 "Read file: auth.ts"
  status: "pending" | "running" | "done" | "error";
  details?: TaskOperation[];  // 底层操作，默认折叠
}

// Agent端需要emit高层步骤事件:
// data: {"taskStep":{"label":"分析代码结构","status":"running"}}\n\n
```

## 八、成功标准

| 指标 | 当前 | 目标 |
|------|------|------|
| 语音交互完成率（说完话到收到回复） | ~60% (需要打开面板) | >95% (悬浮球原位完成) |
| 语音到首字延迟 | ~2-3s | <1s |
| 日均语音交互次数 | ~2次/人 | >10次/人 |
| 专业模式任务理解度 | 低（显示底层操作） | 高（显示任务计划） |
| 首次使用5分钟内完成一次语音交互 | 未测量 | >80% |

## 九、风险与依赖

| 风险 | 影响 | 缓解 |
|------|------|------|
| Tauri窗口动态resize可能有闪烁 | 语音模式体验 | 用CSS动画过渡+预设几个固定尺寸 |
| TTS分句播报延迟 | 语音流畅度 | AudioQueuePlayer已有buffer机制 |
| 后端需要emit高层任务事件 | 专业模式分层展示 | 先前端硬编码映射规则，后续后端配合 |
| heylemon.ai用fn键唤醒，Windows无fn键 | 快捷键设计 | 用Ctrl+Space或其他低冲突组合键 |

---

*总结: 桌面端需要从"什么都想做的单一面板"重构为"语音优先+按需切换专业模式"的双模架构。Phase 1 聚焦语音模式做到 heylemon.ai 水准，Phase 2 强化专业模式做到真正可用。*
