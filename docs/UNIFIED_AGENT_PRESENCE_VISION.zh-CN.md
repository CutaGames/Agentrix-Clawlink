# Agentrix Unified Agent Presence 愿景

生成时间：2026-03-16

## 1. 一句话定义

Agentrix 要做的不是“多个渠道接入多个 Bot”，而是“让用户拥有多个持续在线的 Agent 分身，每个 Agent 都可以跨设备、跨应用、跨社交渠道与用户协作”。

## 2. 这个目标意味着什么

### 多个用户自主管理的 Agent

- 用户可以拥有多个 Agent
- 每个 Agent 可以有不同人格、不同记忆、不同职责
- 用户可以安排某个 Agent 专门负责市场，某个负责开发，某个负责运营，某个负责生活事务
- 每个 Agent 都是长期存在的数字分身，而不是一次性会话

### 每个 Agent 都具备跨端 Presence

- 手机里是它
- 桌面端里是它
- Telegram 里是它
- 飞书里是它
- 企业微信里是它
- 未来耳机、穿戴设备、车载语音里也是它

这里的关键不是“用户只有一个 Agent”，而是“用户可以有多个 Agent，但每个 Agent 在不同入口里都要保持同一个身份和连续性”。

### 同一份上下文

- 任一入口收到的消息，都应该进入统一时间线
- 任一入口形成的结论，都应该写回统一记忆
- 任一入口触发的任务，都应该更新统一状态

但这个统一默认是以单个 Agent 为边界：

- 市场 Agent 有自己的会话、记忆、任务和风格
- 开发 Agent 有自己的会话、记忆、任务和风格
- 是否跨 Agent 共享，必须由用户自己决定

### 同一份体验

- 用户不需要在不同平台重新介绍自己
- 用户不需要在不同平台重复解释任务背景
- 用户不需要记住不同 Bot 的不同行为逻辑

## 3. 产品三层结构

### 第一层：Agent Core

这是核心层：

- 身份
- 记忆
- 任务状态
- 偏好
- 权限

这层还应补一个关键原则：

- 每个 Agent 都有自己的 Core
- 不同 Agent 的 Core 默认彼此独立
- 是否共享记忆、知识库、技能、联系人和任务上下文，必须由用户显式控制

### 第二层：Presence Surface

这是 Agent 出现的外部界面：

- 移动端
- 桌面端
- Telegram
- 飞书
- 企业微信
- Discord
- Twitter/X
- 未来更多应用与硬件

### 第三层：Policy & Delegation

这是控制层：

- 什么能自动做
- 什么必须审核
- 什么能语音处理
- 什么能代表用户发言
- 什么绝不能自动执行
- 哪些 Agent 之间允许共享记忆
- 哪些 Agent 之间允许共享知识库
- 哪些 Agent 之间允许共享技能
- 哪些 Agent 之间完全隔离

## 4. 为什么“社交接入”必须按这个目标重写

如果把社交渠道只当成客服接口，产品会天然走偏：

- 会变成多个孤立 Bot
- 会变成多个孤立消息池
- 会变成多个孤立配置页
- 用户会感觉自己在管理系统，而不是在使用一个 Agent

如果误写成“用户只能有一个 Agent”，同样会走偏：

- 无法支持用户按职责拆分 Agent
- 无法支持不同人格和不同记忆边界
- 无法支持用户精细控制 Agent 之间的共享关系

而真正正确的方向是：

- 渠道只是 Agent 的外部触点
- Bot 只是 Agent 的一个外显身份
- 重点不在“接了多少渠道”
- 重点在“每个 Agent 是否能持续陪伴用户”
- 重点也在“多个 Agent 是否能在用户控制下形成协作网络”

## 5. 关键能力地图

### A. 统一消息总线

- 聊天消息
- 社交消息
- 系统通知
- 任务事件
- 审批事件

都进入统一事件流。

### B. 统一记忆系统

- 个人偏好
- 常见联系人
- 群组语气
- 历史任务背景
- 关系维护状态

这个系统应升级为“可控共享记忆系统”：

- Agent 私有记忆
- 用户级共享记忆
- 指定 Agent 之间的定向共享记忆
- 只共享摘要，不共享原始上下文
- 完全隔离

### C. 语音优先能力

- 消息摘要朗读
- 语音回复
- 语音确认发送
- 语音加入待办
- 语音切换处理优先级

### D. 代理参与能力

- 自动草稿
- 自动回复
- 定时维护
- 社群轻运营
- 跟进提醒

### E. 用户掌控的 Agent 编排能力

- 创建多个不同职责的 Agent
- 为每个 Agent 分配不同技能组合
- 为每个 Agent 指定不同记忆范围
- 控制 Agent 之间是否共享知识库和技能
- 控制 Agent 之间是否能互相协作

## 6. 设备角色分工

### 移动端

- 最随身
- 最适合即时语音
- 最适合免看屏快速处理
- 最适合在多个 Agent 之间快速切换

### 桌面端

- 最适合复杂任务
- 最适合长期在线
- 最适合批量运营和深度上下文处理
- 最适合管理多个 Agent 的边界、共享关系和技能编排

### 社交 Bot

- 最适合外部应用常驻
- 最适合让用户不打开 Agentrix 也继续和某个指定 Agent 协作

## 7. 推荐的落地顺序

1. 先把 Telegram 做成第一个统一 Presence 入口
2. 打通社交消息与主聊天统一时间线
3. 先打通单个 Agent 维度的记忆共享与任务状态共享
4. 打通语音摘要与语音确认发送
5. 再补用户控制的 Agent 间共享策略
6. 再复用到飞书、企业微信、Discord、Twitter/X

## 8. 产品判断标准

以后每做一个新入口，都应该问 5 个问题：

1. 这是不是用户选定的某个 Agent，而不是新 Bot
2. 这条消息会不会进入这个 Agent 的统一时间线
3. 这次互动会不会按用户设定写入该 Agent 或共享记忆
4. 用户能不能自然切换到别的设备继续处理
5. 这个入口是否支持适当的语音和代理能力

如果 5 个问题里有 3 个以上回答是否定的，那就说明方案还没有真正符合 Agentrix 的总目标。
